# BudgetWise landing page with email capture (v2)

This change replaces the default Next.js 16 starter with a five-component BudgetWise landing page: Header, Hero, Features, EmailSignup, and Footer. Components are composed in `page.tsx` via semantic HTML landmarks, themed with a green/teal Tailwind CSS 4 brand palette, and keep hydration cost to a single client boundary (the email form). All four issues from the v1 review have been addressed: the dark-mode media query is removed, Hero has `aria-labelledby`, the form includes a `<noscript>` fallback, and the simulated submission now fails 15% of the time.

**Watch for:** The `aria-describedby="email-status"` on the email input references a status div that persists after the form is replaced by the success message, leaving a 24px empty space below the success state and an orphaned aria reference (confirmed). The Footer renders `new Date().getFullYear()` at build time in a server component, which will go stale if the static artifact is served across a year boundary without rebuild (likely, low severity).

## High-level view

All v1 concerns are resolved cleanly.

The `#email-status` div lives outside the form/success ternary. When the form disappears on success, the status div remains — empty but still taking up `min-h-[1.5rem]` of vertical space beneath the success message. Moving it inside the form branch would fix both the layout gap and the structural orphan.

The Footer's `new Date().getFullYear()` is evaluated at build time under static generation. The year goes stale if the artifact outlives December 31 without a redeploy.

## Aria-describedby lifecycle in EmailSignup

The `#email-status` div sits outside the ternary that switches between the form and the success message:

```tsx
{status === "success" ? (
  <div role="status" ...>...</div>
) : (
  <form ...>
    <input aria-describedby="email-status" ... />
    ...
  </form>
)}

<div id="email-status" aria-live="polite" aria-atomic="true" className="mt-3 min-h-[1.5rem]">
  {status === "error" && <p ...>{errorMessage}</p>}
</div>
```

When `status` is `"success"`, the input is gone but the status div persists — empty, with `min-h-[1.5rem]` creating a 24px gap below the success message. Moving the status div inside the form branch eliminates both the orphaned reference and the visual gap in the success state.
