# BudgetWise landing page with email capture

This change replaces the default Next.js 16 starter with a five-component landing page for BudgetWise: Header, Hero, Features, EmailSignup (client component), and Footer. The page is composed via semantic landmarks (`<header>`, `<main>`, `<footer>`), uses Tailwind CSS 4 custom theme tokens for a green/teal brand palette, and keeps all content server-rendered except the interactive email form. The email form simulates submission with a timeout, provides validation feedback via aria-live regions, and transitions through idle/loading/success/error states.

**Watch for:** Dark mode is partially wired — the CSS swaps `--background`/`--foreground` via a `prefers-color-scheme: dark` media query, but every component hardcodes light-only classes like `bg-white`, `text-neutral-900`, `bg-primary-50` without `dark:` variants, producing an unreadable page in dark environments (confirmed). The Hero section lacks `aria-labelledby` unlike its sibling sections, leaving it anonymous to screen readers (confirmed). The simulated email submission always succeeds — the server/network error UI path is unreachable and untestable (confirmed).

## High-level view

Five leaf components composed in `page.tsx`: Header, Hero, Features, and Footer are server components; EmailSignup is the sole client boundary, keeping hydration cost limited to the form.

The design system defines `--color-primary-*` and `--color-neutral-*` tokens in a `@theme inline` block. The palette is light-mode only, but a dark-mode media query in the same CSS swaps body-level variables that nothing else references — the page breaks visually on dark-mode systems.

The email form has no `action` attribute, so it submits nowhere without JavaScript. The PRD permits this ("form submission may require JS"), but users without JS see a form with no indication it won't work. A `<noscript>` disclaimer would close this gap.

The Hero section is the only `<section>` without `aria-labelledby`, making it the sole anonymous landmark when navigating by region.

## Dark mode incompatibility

The `globals.css` file defines a `prefers-color-scheme: dark` media query that swaps `--background` to `#0a0a0a` and `--foreground` to `#ededed`. The `body` rule applies these variables. But every component uses fixed Tailwind classes: `bg-white` on Header and Footer, `bg-primary-50` on Hero and EmailSignup, `text-neutral-900` on all headings, `bg-neutral-50` on feature cards. None of these respond to dark mode.

On a dark-mode system, the body background turns near-black while the Header sits as a bright white bar, headings remain dark-on-dark (illegible against `bg-primary-50` which is a very light green), and the overall contrast is broken. If dark mode isn't intended for MVP, remove the media query to avoid signaling support that doesn't exist. If it is intended, every component needs `dark:` variants.

## Hero section landmark gap

Features uses `aria-labelledby="features-heading"`, EmailSignup uses `aria-labelledby="signup-heading"`, but Hero has neither an `id` on its heading nor an `aria-labelledby` on its section. A screen reader navigating by landmarks will identify Features and EmailSignup by name but the Hero section remains anonymous. Adding `id="hero-heading"` to the `<h1>` and `aria-labelledby="hero-heading"` to the `<section>` fixes this.

## Email form progressive enhancement

The `<form>` element has `onSubmit={handleSubmit}` and `noValidate`, but no `action` attribute. With JavaScript disabled, the submit button is inert — no page navigation, no POST. The user sees a fully-rendered form with no indication it won't work. A `<noscript>` element inside or near the form section saying "JavaScript is required to join the waitlist" would be a minimal fix that's transparent about the dependency.

## Submission simulation always succeeds

In `EmailSignup.tsx`, the form transitions from `loading` to `success` unconditionally:

```tsx
setTimeout(() => {
  setStatus("success");
}, 1500);
```

The error state is only reachable via the client-side email format check. The "server error" styling and messaging will be exercised for the first time when a real backend replaces the simulation. Adding a small probability of simulated failure (or a `simulateError` prop) would make the error UI verifiable during development.
