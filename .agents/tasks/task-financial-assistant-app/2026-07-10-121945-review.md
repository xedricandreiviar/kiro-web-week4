# Financial assistant app: login, onboarding, and personalized dashboard (v2)

BudgetWise adds a complete post-login flow: simulated login, a 4-step onboarding questionnaire, a personalized dashboard, and a first-visit app tour. All five issues from the v1 review have been addressed — returning users skip onboarding, the tour now uses positional highlighting with `getBoundingClientRect`, the route guard shows a spinner, negative amounts are rejected, and sidebar links either scroll to sections or show a "Coming Soon" badge. The implementation is solid for a localStorage-only MVP.

**Watch for:** Tour highlight positions are computed once at render time and won't track if the user resizes or scrolls underneath the overlay (likely). Login page reads localStorage directly after calling `login()`, creating a race with the context's state update that could misroute returning users under specific timing conditions (possible).

**Verdict**: APPROVED

## High-level view

The returning-user fix in `login/page.tsx` reads localStorage directly rather than waiting for the context update from `login()`. The fix reads the *pre-existing* stored value, which for a returning user already contains onboarding data because `login()` preserves the stored user object when emails match. The logic is sound but brittle: it depends on `login()` never clearing or replacing the stored value before the component's routing check executes.

The app tour now queries the DOM for target elements and positions the tooltip card relative to `getBoundingClientRect()`. The highlight border and card position are computed from `targetRect` on each render triggered by step navigation. Since there's no resize/scroll listener, the highlight rectangle is a snapshot that can drift if the page layout shifts while the tour is active.

<summary>Issues (2)</summary>

1. **Tour highlight position is static** — `getBoundingClientRect()` is called once per step transition. If the user resizes the browser or the page layout shifts (e.g., mobile keyboard, dynamic content), the highlight rectangle drifts away from its target. Add a resize/scroll listener that recalculates position, or use a `ResizeObserver` on the target element.
2. **Login routing depends on localStorage read timing** — The `handleSubmit` in `login/page.tsx` reads localStorage immediately after calling `login()`. This works today because `login()` preserves the existing user object for matching emails. If `login()` ever clears/replaces the stored user before the component re-renders, the routing check would read stale data. A safer pattern: read the return value from `login()` or check `user.onboarding` from context after the state settles.

## Tour positioning without layout tracking

The `AppTour` component computes `targetRect` via `document.getElementById(step.targetId)?.getBoundingClientRect()` on each render. Step transitions trigger a re-render (via `setCurrentStep`), so the position updates when moving between steps. But within a single step, nothing triggers a re-render if the viewport changes.

```tsx
const target = typeof window !== "undefined" ? document.getElementById(step.targetId) : null;
const targetRect = target?.getBoundingClientRect() ?? null;
```

On mobile where orientation changes or soft keyboards shift layout, the highlight box will visually separate from its target. The fallback (centered card) handles the case where the target isn't found, so there's no crash — just a UX drift.

There's also a timing gap between `scrollIntoView` (asynchronous) and `getBoundingClientRect()` (synchronous at call time). If the target is off-screen when a step transition fires, the highlight will briefly point at the pre-scroll position. React's batched state update means the component re-renders after `setCurrentStep`, by which time scroll may not have completed — so the first rendered frame after a step change can show the highlight at the wrong position.

## Login routing logic

```tsx
const handleSubmit = (e: FormEvent) => {
  e.preventDefault();
  login(email, password);
  
  const stored = localStorage.getItem(STORAGE_KEY);
  if (stored) {
    const existingUser = JSON.parse(stored);
    if (existingUser.email === email && existingUser.onboarding) {
      router.push("/dashboard");
      return;
    }
  }
  router.push("/onboarding");
};
```

`login()` in `AuthContext` calls `setUser(existingUser)` for matching emails, preserving the stored object including `onboarding`. The localStorage write happens in a `useEffect` on the next render. So `handleSubmit` reads the *previous* persisted state — which for a returning user is their complete profile.

The fragility: if `AuthContext.login()` were refactored to clear localStorage before setting state (e.g., to handle multi-user switching), this routing logic would break silently. The coupling between the login page's localStorage read and the context's internal persistence timing is implicit.
