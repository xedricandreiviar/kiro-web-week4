# Financial assistant app: login, onboarding, and personalized dashboard

BudgetWise gains a complete post-login user flow: a simulated login page, a 4-step onboarding questionnaire that captures income range, spending categories, financial goals, and savings targets, then a personalized dashboard with budget tracking, expense management, savings progress, and category breakdowns. An app tour overlay fires on first visit to walk the user through the dashboard sections. All state lives in localStorage with React Context as the runtime bridge. No backend involved.

**Watch for:** Login always succeeds with any credentials, and returning users are always re-routed through onboarding (confirmed). The app tour highlights dashboard elements by description but doesn't actually scroll or visually point to those elements — it's a centered modal, not a positional tooltip (confirmed). The dashboard route guard renders nothing during the redirect frame, causing a flash (likely).

## High-level view

The auth layer is intentionally a stub — any email/password combination logs you in and the password is discarded. The login page communicates this only via placeholder text ("Enter any password") that disappears on focus.

Every login — including returning users who already completed onboarding — redirects to `/onboarding` unconditionally. There's no check for whether onboarding data already exists, so personalization is re-asked every session unless the user navigates directly to `/dashboard`.

The app tour is a full-screen modal overlay with step descriptions. It defines `targetId` per step and those IDs exist on dashboard sections, but the component never queries the DOM for them. Each step describes a section conceptually while the modal sits centered — no positional highlighting, no scrolling to the referenced element.

Route protection in the dashboard layout uses a `useEffect` redirect, but returns `null` during the redirect tick, causing a blank flash before navigation fires.

All sidebar navigation links (Expenses, Budget, Goals, Settings) resolve to the same `/dashboard` path, producing no visible effect on click.

<summary>Details</summary>

## Returning users always repeat onboarding

In `app/login/page.tsx`, the submit handler calls `login(email, password)` then unconditionally calls `router.push("/onboarding")`. The onboarding page itself only checks `isLoggedIn` — it doesn't check whether `user.onboarding` is already populated. A returning user whose localStorage still has their profile (including onboarding data) will be forced through the questionnaire again every time they log in.

The fix is straightforward: after calling `login()`, check if the stored user already has onboarding data and route to `/dashboard` if so, `/onboarding` otherwise. Alternatively, the onboarding page could detect existing data and skip ahead.

## App tour: modal without positional guidance

`AppTour.tsx` defines a `targetId` field per step, implying the tour will highlight the corresponding DOM element. In practice, the component never queries the DOM for those IDs. The tour renders as a centered floating card with a semi-transparent backdrop — identical regardless of which step is active.

The mismatch between the tour's descriptive text ("Track your monthly budget versus actual spending at a glance") and the lack of visual pointing may confuse first-time users who can't locate the referenced section behind the overlay.

## Route guard flash

`app/dashboard/layout.tsx`:

```tsx
if (!isLoggedIn) {
  return null;
}
```

This returns an empty DOM tree for the frame between initial render and `useEffect` redirect execution. On slower connections or devices, this manifests as a white flash. A lightweight loading indicator or the existing BudgetWise logo as a placeholder would prevent the layout shift.

## Expense form accepts invalid amounts

The add-expense form in `app/dashboard/page.tsx` uses `type="number"` with `step="0.01"` but no `min` attribute. Submitting a negative value results in a negative expense stored in localStorage. An empty-string edge case produces `NaN`, rendered as "-$NaN" in the expenses list.

Adding `min="0.01"` on the input and a guard in `handleAddExpense`:

```tsx
const amount = parseFloat(newAmount);
if (isNaN(amount) || amount <= 0) return;
```

would close this gap.

## Dead sidebar links

All five items in `NAV_ITEMS` inside `app/dashboard/layout.tsx` resolve to `/dashboard`:

```tsx
const NAV_ITEMS = [
  { label: "Dashboard", href: "/dashboard", icon: "..." },
  { label: "Expenses", href: "/dashboard", icon: "..." },
  { label: "Budget", href: "/dashboard", icon: "..." },
  { label: "Goals", href: "/dashboard", icon: "..." },
  { label: "Settings", href: "/dashboard", icon: "..." },
];
```

Clicking "Expenses" or "Goals" produces no visible change. Either these should scroll to the relevant dashboard section (via anchor links or `scrollIntoView`) or should be rendered as disabled/coming-soon indicators.
