# Comprehensive personal budgeting app with 12 dashboard pages

This change transforms a single-page expense tracker into a full-featured personal finance app: 12 new routes covering transactions, budgets, income, savings goals, bills, debts, net worth, reports, calendar, insights, and settings. All data lives in localStorage via a centralized typed storage layer (`app/lib/storage.ts`). The dashboard overview was refactored to pull from this shared layer, and mock data is seeded on first load based on onboarding preferences.

Watch for: localStorage quota exhaustion has no user-visible feedback (confirmed). The `generateId()` function uses `Date.now()` and is susceptible to collisions under rapid-fire operations (likely). No data import to complement the export feature (confirmed). Overdue bill detection resets each month, missing carry-over from prior months (confirmed).

## High-level view

The storage layer silently drops writes when quota is exceeded (`safeSetItem` catches and ignores errors), meaning a user could lose a transaction without any indication. Since every mutation re-reads and re-writes the full entity array, this failure mode can trigger whenever the aggregate data across all seven entity types approaches the ~5MB localStorage limit.

ID generation uses `Date.now()` concatenated with a 9-char random suffix. The seed function calls this ~28 times in a tight synchronous loop, and `Date.now()` can return the same value across those calls. Collision probability per seed invocation is low but non-zero; `crypto.randomUUID()` would eliminate the concern.

The bills page `isOverdue` function compares only the current day-of-month against the bill's due day. A bill due on the 28th that was never paid last month will appear non-overdue on days 1–27 of the new month. The `getNextDueDate` display logic is more sophisticated, but the actual overdue flag (which drives the red highlighting) uses the simpler check.

The net worth page pulls debts from `debtStorage` and shows them alongside manually-entered liabilities. Nothing prevents a user from entering the same debt in both places, silently double-counting liabilities.

The calendar maps recurring income events to a fixed day-of-month derived from the transaction's original date. Biweekly or weekly frequencies don't land on a consistent day-of-month, so those entries appear on the wrong dates across months.

## Storage layer and ID generation

`generateId` in `utils.ts`:

```ts
export function generateId(): string {
  return `${Date.now()}-${Math.random().toString(36).substring(2, 11)}`;
}
```

The seed function in `seed.ts` calls `generateId()` in a tight loop (18+ times for transactions, 4 for budgets, 2 for goals, 4 for bills — ~28 sequential calls). `Date.now()` returns the same value for all calls within the same millisecond, so collision avoidance rests entirely on the 9-character random suffix (~10^14 namespace). The probability is low per pair but across 28 calls in a single invocation the birthday-problem math puts it around 1 in 10^7. `crypto.randomUUID()` eliminates this class entirely and is available in all target environments.

## Overdue logic in bills page

```ts
function isOverdue(bill: Bill): boolean {
  if (bill.isPaid) return false;
  const today = new Date();
  today.setHours(0, 0, 0, 0);
  const currentDay = today.getDate();
  return currentDay > bill.dueDate;
}
```

This doesn't account for month boundaries. A bill due on the 28th that wasn't paid last month appears non-overdue on the 1st–27th of the following month. The bill's `lastPaidDate` field exists in the interface but isn't consulted here — comparing it against the expected due date would fix the false-negative window.

## Debt payoff strategy calculation

The debts page computes avalanche and snowball strategies on every render via inline state derivation (not memoized):

```ts
const avalanche = calculateStrategy(debts, "avalanche");
const snowball = calculateStrategy(debts, "snowball");
```

`calculateStrategy` iterates up to 600 months in a while-loop, applying interest and payments to each debt per iteration. With 5 debts, that's 3000 inner-loop iterations in the worst case. Wrapping in `useMemo` keyed on `debts` would prevent recomputation when form state or payment modals toggle.

## Net worth double-counting

The net worth page pulls debts from `debtStorage.getAll()` and displays them as read-only liabilities alongside user-entered liabilities. If a user adds a "Student Loan" debt on the Debts page and also manually adds a "Student Loan" liability on the Net Worth page, both appear in the liabilities total. The UI gives no indication that debts are already included or that manual liabilities should exclude them.

## Calendar frequency mismatch

```ts
recurringIncome.forEach((t) => {
  const tDate = new Date(t.date);
  if (tDate.getDate() === day) {
    events.push({ type: "income", name: t.description, amount: t.amount });
  }
});
```

A salary recorded on January 15 will appear on the 15th of every month regardless of whether its `recurringFrequency` is `"biweekly"` or `"weekly"`. The calendar treats all recurring income as monthly by pinning to day-of-month, which is incorrect for non-monthly frequencies.

## Test coverage

There are no test files in this diff. The 12 new pages, storage layer, utility functions, and seed logic have zero automated tests.

Not tested: storage CRUD edge cases (quota overflow, corrupt JSON), ID uniqueness under rapid generation, debt payoff calculation correctness, calendar event placement for non-monthly frequencies, insights generation thresholds, data export format integrity, overdue bill detection across month boundaries.
