# Comprehensive personal budgeting app (v2)

This change builds a 12-page personal finance dashboard on top of a typed localStorage layer. The v2 fixes address all six concerns raised in the first review: storage now returns success/failure from writes, ID generation uses `crypto.randomUUID()` with fallback, data import was added to settings, the debt payoff calculation is memoized, overdue bill detection considers `lastPaidDate` and cross-month logic, and the net worth page shows a warning about debt auto-inclusion. The calendar also received a proper frequency-aware implementation for weekly and biweekly recurring income.

Watch for: Storage write failures are surfaced to the caller but no page actually checks the return value or shows user feedback (confirmed). The import function bypasses the storage layer and writes directly via `localStorage.setItem`, risking partial state on quota failure with a misleading error message (confirmed). Calendar event computation runs inside the render loop for every cell on every render without memoization (confirmed).

## High-level view

The storage layer now returns `{ data, success }` from `create` and `update`, but every dashboard page ignores the `success` field. The quota feedback problem was half-solved: the infrastructure exists, but the user never sees a warning because no caller checks the return.

The import feature in Settings validates JSON structure and writes each entity type via raw `localStorage.setItem` inside a try/catch. A `QuotaExceededError` partway through the sequence will be caught, but the error message says "Failed to parse the file" â€” misleading when the parse succeeded but the write failed. More critically, entity types written before the failure remain, leaving the database in a partial state with no rollback.

The calendar handles weekly and biweekly frequencies correctly now, but `getEventsForDay` is called per visible cell (up to 42 times) on every render, iterating all entities each time. Pre-computing events by day for the visible month would collapse this to a single pass.

## Storage success signaling without consumer

The fix adds a boolean return from `safeSetItem` and propagates it through `createEntity` and `updateEntity` as `{ data, success }`. But every page consumes it like:

```ts
billStorage.create(bill);       // return value discarded
setBills(billStorage.getAll()); // re-reads, will miss the failed write
```

The re-read after a failed write means the UI won't show the un-persisted entity on reload, but the user gets no immediate indication that their action was lost.

## Import writes raw localStorage and risks partial state

```ts
if (Array.isArray(data.transactions)) {
  localStorage.setItem("budgetwise_transactions", JSON.stringify(data.transactions));
}
if (Array.isArray(data.budgets)) {
  localStorage.setItem("budgetwise_budgets", JSON.stringify(data.budgets));
}
// ... 5 more entity types
```

These calls are inside a try/catch, so a `QuotaExceededError` thrown mid-sequence (say, after transactions and budgets wrote but before debts) is caught â€” but the handler shows "Failed to parse the file. Make sure it is a valid JSON export." This is misleading: the file parsed fine, the write failed. The partial state (some entities imported, others not) persists with no rollback or indication of what succeeded.

## Calendar renders O(cells * entities) per frame

The grid renders each cell by calling `getEventsForDay(day, ...)` inline:

```tsx
{calendarDays.map((day, idx) => {
  const events = getEventsForDay(day, currentMonth, currentYear, data.bills, recurringIncome, data.goals);
```

With 42 cells and 26 entities, that's ~1092 iterations per render. The function creates `new Date()` objects and computes modular arithmetic per income entry per cell. A `useMemo` that pre-computes events keyed by day for the visible month would collapse this to a single pass.
